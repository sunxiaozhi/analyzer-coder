# 研发知识资产管理平台需求文档

## 1. 产品定位

本项目从 Java 代码分析器演进为面向研发团队的知识资产管理平台。

产品核心不是再做一个代码图谱工具，而是管理团队人工总结、人工确认、可复审、可追踪的研发知识。代码分析、接口映射、向量索引和语义检索作为证据层，为知识资产提供关联代码、接口、文件和搜索上下文。

一句话定位：

```text
把业务规则、架构决策、故障经验、接口说明和代码证据绑定起来，形成团队可信、AI 可用、持续复审的项目记忆。
```

与 CodeGraph 类工具的差异：

```text
CodeGraph 主要回答：代码在哪里、谁调用谁、改动影响什么。
本产品主要回答：团队确认过什么知识、为什么这么做、证据在哪、是否仍然可信。
```

## 2. 当前产品能力总览

当前系统已经具备以下能力：

- 用户登录、管理员和项目成员权限。
- 项目管理，支持通过 Git 地址创建项目、拉取更新，并按项目隔离数据。
- Java 静态分析，提取类型、字段、方法、调用点、Spring 组件、HTTP endpoint、MyBatis SQL 注解。
- 分析报告生成，支持 Markdown 报告、JSON 结果、Mermaid 图。
- 前后端 API 映射，识别前端请求和后端接口的匹配关系。
- 知识文件维护，支持 Markdown、TXT、RST、AsciiDoc 等文档的创建、编辑、删除。
- 知识模板维护，支持业务规则、接口说明、决策记录、故障排查等模板。
- 知识资产管理，支持结构化管理人工总结的业务规则、ADR、故障案例、接口说明、开发规范、领域术语和模块说明。
- 向量索引，支持代码 chunk、知识文档 chunk、知识资产 chunk 的统一索引。
- 语义检索，支持代码、知识文档、知识资产的检索和关联证据展示。
- Web 控制台，使用 Flask 后端和 Vue 3 前端承载上述能力。

## 3. 用户角色

### 3.1 管理员

管理员负责平台配置、账号管理和项目接入。

核心操作：

- 创建项目。
- 拉取项目代码。
- 创建账号。
- 分配用户可访问项目。
- 查看和维护全部项目下的知识资产。
- 确认或标记知识资产状态。

### 3.2 项目成员

项目成员负责维护项目知识和使用检索能力。

核心操作：

- 查看被授权项目。
- 维护项目知识文件。
- 创建和编辑知识资产。
- 关联代码证据。
- 重建索引。
- 使用语义检索查询知识和代码证据。

### 3.3 AI 编程助手使用者

AI 编程助手使用者关注的是将项目知识作为上下文输入给 Codex、Claude Code、Cursor 等工具。

核心诉求：

- 快速找到当前任务相关的已确认知识。
- 查看知识背后的代码证据。
- 识别过期或待复审知识，避免 AI 采信错误上下文。
- 将检索结果作为任务上下文提供给 AI。

## 4. 核心业务闭环

### 4.1 项目接入闭环

```text
管理员创建项目
-> 系统 clone 仓库到项目数据目录
-> 用户选择项目
-> 后续分析、知识、索引、检索都在该项目范围内执行
```

对应当前功能：

- `POST /api/projects`
- `GET /api/projects`
- `POST /api/projects/<project_id>/pull`
- 前端：项目管理页面。

### 4.2 代码分析闭环

```text
选择项目
-> 执行代码分析
-> 提取 Java 结构、Spring endpoint、组件和调用关系
-> 生成报告、JSON 或 Mermaid 图
-> 分析结果作为知识证据来源
```

对应当前功能：

- `POST /api/analyze`
- `GET /api/analysis/result`
- Java 分析器：`src/java_analyzer/analyzer.py`
- 前端：代码分析页面。

### 4.3 知识沉淀闭环

```text
项目成员根据业务、故障、接口、规范沉淀知识
-> 使用模板或手动编写知识内容
-> 创建结构化知识资产
-> 设置类型、状态、负责人、标签和复审时间
-> 挂载代码证据
-> 确认后成为团队可信知识
```

对应当前功能：

- `GET /api/knowledge/assets`
- `POST /api/knowledge/assets`
- `PUT /api/knowledge/assets/<asset_id>`
- `DELETE /api/knowledge/assets/<asset_id>`
- `POST /api/knowledge/assets/<asset_id>/confirm`
- `POST /api/knowledge/assets/<asset_id>/mark-stale`
- 前端：知识资产页面。

### 4.4 文档维护闭环

```text
创建知识文档
-> 编辑 Markdown / TXT / RST / AsciiDoc 内容
-> 按项目隔离保存
-> 作为知识库文档参与索引
```

对应当前功能：

- `GET /api/kb/files`
- `GET /api/kb/file`
- `POST /api/kb/files`
- `PUT /api/kb/file`
- `DELETE /api/kb/file`
- 前端：知识维护页面。

### 4.5 模板复用闭环

```text
维护知识模板
-> 创建业务规则、接口说明、故障案例、ADR 等模板
-> 知识维护时基于模板生成文档
-> 降低团队沉淀知识的成本
```

对应当前功能：

- `GET /api/kb/templates`
- `POST /api/kb/templates`
- `PUT /api/kb/template`
- `DELETE /api/kb/template`
- 前端：知识模板页面。

### 4.6 索引与检索闭环

```text
选择项目
-> 建立代码、知识文档、知识资产的混合索引
-> 用户输入问题
-> 系统返回直接命中结果
-> 同时生成相关代码和相关知识证据
-> 用户判断哪些知识可作为 AI 上下文
```

对应当前功能：

- `POST /api/index`
- `GET /api/index/status`
- `GET /api/index/records`
- `POST /api/query`
- 前端：索引运维页面、知识检索页面。

### 4.7 接口知识闭环

```text
执行 API 映射
-> 识别前端调用和后端 endpoint 的匹配关系
-> 暴露未匹配或方法不一致的接口
-> 为接口说明类知识资产提供候选证据
```

对应当前功能：

- `POST /api/api-map`
- 前端：接口映射页面。

## 5. 功能模块需求

### 5.1 登录与账号管理

已实现能力：

- 用户登录。
- 用户退出。
- 当前用户查询。
- 管理员创建用户。
- 管理员编辑用户基础信息。
- 管理员修改用户密码。
- 管理员配置用户项目访问权限。
- 保存用户最后选择项目。

需求要求：

- 非登录用户只能看到登录页。
- 普通用户不能访问账号管理。
- 普通用户只能看到被授权项目。
- 管理员可以看到全部项目和全部用户。
- 系统必须至少保留一个管理员。

### 5.2 项目管理

已实现能力：

- 创建项目时输入项目名称、Git 地址和可选分支。
- 系统 clone 项目到本地项目数据目录。
- 项目数据按项目隔离。
- 支持拉取项目最新代码。

需求要求：

- 项目创建成功后自动授权给创建人。
- 项目路径必须限制在配置的 workspace/project 根目录内。
- Git 操作失败时需要返回明确错误。
- 不允许项目数据串项目访问。

### 5.3 知识资产管理

知识资产是本产品的核心对象。

已实现能力：

- 知识资产列表。
- 按类型、状态、关键词筛选。
- 创建知识资产。
- 编辑知识资产。
- 删除知识资产。
- 确认知识资产。
- 标记知识资产为待复审。
- 维护负责人、复审日期、标签、来源文档和代码证据。
- 知识资产进入统一向量索引，`source_type` 为 `knowledge_asset`。

知识资产类型：

| 类型 | 说明 |
|---|---|
| `business_rule` | 业务规则 |
| `adr` | 架构决策 |
| `incident` | 故障案例 |
| `api_doc` | 接口说明 |
| `standard` | 开发规范 |
| `glossary` | 领域术语 |
| `module_note` | 模块说明 |

知识资产状态：

| 状态 | 说明 |
|---|---|
| `draft` | 草稿 |
| `pending_review` | 待确认 |
| `confirmed` | 已确认 |
| `stale` | 待复审 |
| `archived` | 已归档 |

需求要求：

- 新建资产默认状态为 `draft`。
- 资产确认后记录确认人和确认时间。
- 标记待复审后，检索和页面应明显展示其非最终可信状态。
- 知识资产必须有标题。
- 知识资产必须属于一个项目。
- 证据可以为空，但已确认资产建议至少有一条证据。
- 后续应支持从分析结果、API 映射、索引记录中一键添加证据。

### 5.4 知识文件维护

已实现能力：

- 列出知识库文件。
- 打开知识库文件。
- 创建知识库文件。
- 保存知识库文件。
- 删除知识库文件。
- 支持 `.md`、`.markdown`、`.txt`、`.rst`、`.adoc`。

需求要求：

- 文件路径必须是相对路径。
- 文件必须限制在项目知识库根目录内。
- 不允许编辑不支持的文件类型。
- 删除文件后，应清空当前编辑状态。

### 5.5 知识模板维护

已实现能力：

- 默认内置业务规则、接口说明、决策记录、故障排查模板。
- 可新增模板。
- 可编辑模板。
- 可删除模板。
- 模板按项目隔离保存。

需求要求：

- 模板必须包含名称、默认路径和内容。
- 模板默认路径必须是相对路径。
- 模板默认路径扩展名必须属于知识库支持格式。
- 后续模板应和知识资产类型绑定，例如业务规则模板默认创建 `business_rule` 类型资产。

### 5.6 代码分析

已实现能力：

- 分析 Java 文件或目录。
- 提取 package、import、类型、字段、方法、构造器、调用点。
- 提取 Spring 组件。
- 提取 HTTP endpoint。
- 提取 MyBatis SQL 注解。
- 生成 Markdown 报告。
- 生成 JSON 结果。
- 生成 Mermaid 图。

需求要求：

- 分析路径必须限制在项目根目录内。
- 分析结果必须按项目隔离保存。
- 代码分析输出应能作为知识资产证据来源。
- 后续应支持增量分析和变更检测。

### 5.7 API 映射

已实现能力：

- 扫描前端 API 调用。
- 扫描后端 endpoint。
- 匹配前端调用和后端接口。
- 标记匹配、方法不一致、未匹配。
- 保存映射结果。

需求要求：

- 映射结果应支持作为 `api_doc` 知识资产的证据。
- 未匹配接口应提示用户补充接口说明或检查代码。
- 后续应支持从映射结果一键创建接口说明资产。

### 5.8 向量索引

已实现能力：

- 代码 chunk 索引。
- 知识文档 chunk 索引。
- 知识资产 chunk 索引。
- 混合索引。
- 索引状态统计。
- 索引记录分页查看。
- 按来源、类型和关键词过滤索引记录。

当前来源类型：

| `source_type` | 说明 |
|---|---|
| `code` | 代码分析 chunk |
| `kb` | 知识库文档 chunk |
| `knowledge_asset` | 结构化知识资产 chunk |

需求要求：

- 默认混合索引应包含代码、知识文档和知识资产。
- 已确认知识资产在检索展示中应优先成为可信知识。
- 待复审和草稿资产可以被检索，但需要展示状态。
- 后续应支持删除失效 chunk。

### 5.9 语义检索

已实现能力：

- 输入自然语言问题进行检索。
- 支持按代码、知识文档、知识资产过滤。
- 返回命中结果。
- 返回关联代码证据。
- 返回关联知识证据。
- 返回结果间共享词关系。

需求要求：

- 检索结果必须展示来源、类型、位置和分数。
- 知识资产命中时必须展示资产状态。
- 检索结果应区分直接命中和关联证据。
- 后续应提供“生成 AI 上下文包”能力，将已确认知识和必要代码证据组织成可复制上下文。

## 8. 非功能需求

### 8.1 安全

- 所有项目路径和知识库路径必须限制在配置的 workspace/project 根目录内。
- 非登录用户不能访问业务 API。
- 普通用户不能访问未授权项目。
- 管理员能力必须进行后端校验。
- 用户密码必须哈希存储。

### 8.2 可部署性

- 开发环境可使用 Flask 开发服务器和 Vite。
- 生产环境应使用 Nginx + Gunicorn + systemd。
- 后端默认支持 MySQL 状态存储。
- 本地开发和测试可使用 JSON 状态存储。

### 8.3 可维护性

- 后端 API 路由集中在 `web/backend/routes.py`。
- 后端业务逻辑集中在 `web/backend/services.py`。
- 前端共享状态集中在 `web/frontend/src/composables/useAnalyzerConsole.ts`。
- 前端复用组件放在 `web/frontend/src/components/analyzer`。
- 共享类型放在 `web/frontend/src/types.ts`。

### 8.4 性能

- 当前索引使用本地 JSONL，适合小型和中型项目验证。
- 后续企业部署建议迁移到 Qdrant 或同类向量库。
- 代码分析和索引可能耗时，应继续保持前端 loading 状态和后端超时控制。

## 9. 当前验收标准

### 9.1 项目可启动

- 后端可通过 `python -m web.backend.app` 启动。
- 前端可通过 `npm run dev` 启动。
- `GET /api/health` 返回 `ok=true`。

### 9.2 知识资产可用

- 用户选择项目后可以进入知识资产页面。
- 可以创建知识资产。
- 可以编辑标题、类型、状态、摘要、正文、标签、负责人、证据。
- 可以确认资产。
- 可以标记资产待复审。
- 可以删除资产。
- 资产按项目隔离保存。

### 9.3 索引可用

- 执行混合索引后，索引中包含代码、知识文档和知识资产。
- 索引状态可以统计不同来源数量。
- 索引记录可以查看 `knowledge_asset` 来源。

### 9.4 检索可用

- 输入查询后可以返回结果。
- 结果可以包含代码、知识文档和知识资产。
- 知识资产作为知识侧参与关联证据构建。

## 10. 后续迭代路线

### 10.1 第一阶段：知识资产 MVP 完善

- 从 API 映射结果一键创建 `api_doc` 知识资产。
- 从分析结果一键添加 Java 类、方法、endpoint 证据。
- 检索页面突出展示 `confirmed` 知识资产。
- 索引运维页面增加 `knowledge_asset` 来源筛选显示优化。
- 知识资产支持复制为 AI 上下文。

### 10.2 第二阶段：知识治理

- 增加知识复审提醒。
- 根据代码变更自动标记关联知识为待复审。
- 增加知识冲突检测。
- 增加确认流：草稿 -> 待确认 -> 已确认。
- 增加资产历史版本。

### 10.3 第三阶段：企业级存储与检索

- 知识资产从 JSON 文件迁移到 MySQL 表。
- 证据关系从 JSON 字段迁移到独立关系表。
- JSONL 向量库迁移到 Qdrant。
- 支持增量索引和删除失效 chunk。
- 支持按分支、commit、模块维度过滤。

### 10.4 第四阶段：AI 上下文产品化

- 生成任务上下文包。
- 区分可信知识、待复审知识和普通代码证据。
- 为 Codex、Claude Code、Cursor 提供可复制上下文。
- 后续可提供 MCP 接口，但主产品仍以人工知识治理为中心。

## 11. 成功指标

建议从以下指标衡量产品是否跑通：

- 每个项目沉淀的已确认知识资产数量。
- 每条已确认资产平均关联证据数量。
- 待复审资产平均处理时长。
- 检索结果中已确认知识资产命中比例。
- 新人或 AI 任务中复用知识资产的次数。
- 因知识过期导致的问题数量是否下降。

## 12. 当前边界

当前版本不追求替代 CodeGraph，不追求完整多语言调用图，不追求复杂图数据库。

当前版本重点是：

```text
把已经能分析出的 Java 代码事实、知识文档和人工知识资产串成一个可用闭环。
```

优先保证团队能持续沉淀、确认、检索和复审知识，再逐步增强代码证据自动化和 AI 上下文输出能力。
