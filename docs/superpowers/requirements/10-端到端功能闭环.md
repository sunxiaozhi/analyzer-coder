# 端到端功能闭环需求

## 功能目标

本文件描述平台从项目接入到知识治理的完整业务闭环，确保拆分后的功能可以串联为一个可落地的研发智能工作流。

端到端闭环要解决的问题：

- 用户进入项目后，能从代码事实和团队知识中获得可信上下文。
- 检索结果、上下文包、影响分析和治理任务能复用同一套项目数据。
- 知识和证据发生变化后，能回流到索引、图关系、检索和上下文包。
- 系统在任务失败、低命中、证据失效时有明确兜底和治理入口。

## 总体业务链路

```text
项目接入
  -> 仓库同步
  -> 代码分析
  -> 代码事实入库
  -> Neo4j 基础图关系写入
  -> 知识文档和知识资产维护
  -> Evidence 绑定
  -> Qdrant 统一索引构建
  -> 可信检索
  -> AI 上下文包生成和导出
  -> 变更影响分析
  -> ReviewTask 或治理项
  -> 知识/Evidence 更新
  -> 标记索引 stale
  -> 更新 Neo4j 图关系
  -> 重新索引
  -> 影响后续检索、上下文包和影响分析
```

## 阶段输入输出

| 阶段 | 输入 | 输出 | 后续消费者 |
|---|---|---|---|
| 项目接入 | Git 地址、分支、默认路径、项目负责人 | Project、ProjectMember、RepositorySnapshot | 仓库同步、权限校验、审计 |
| 仓库同步 | Project、Git 配置 | 本地仓库快照、同步状态 | 代码分析 |
| 代码分析 | RepositorySnapshot、代码路径 | CodeFact、endpoint、SQL、调用关系、失败文件 | Neo4j、Evidence、索引、影响分析 |
| 图关系写入 | CodeFact、Evidence、KnowledgeAsset | Neo4j 节点边、路径查询能力 | 可信检索解释、上下文包、影响分析 |
| 知识沉淀 | 知识文档、人工输入、检索结果、代码事实 | KnowledgeAsset、Evidence | 索引、图关系、上下文包、治理 |
| 统一索引 | CodeFact、知识文档、KnowledgeAsset、Evidence | IndexRecord、Qdrant point | 可信检索、上下文召回、影响分析语义候选 |
| 可信检索 | 查询文本、项目过滤、来源过滤 | SearchResult、SearchLog、可复用结果 | 上下文包、Evidence 绑定、知识草稿、治理 |
| 上下文包 | 任务描述、候选上下文、人工选择 | ContextPackage、引用清单、导出审计 | AI 编程助手、审计、后续复盘 |
| 影响分析 | 文件列表或 diff、CodeFact、Neo4j、Qdrant | ChangeAnalysisTask、候选影响、测试建议、ReviewTask 候选 | 治理复审、知识更新 |
| 治理复审 | 低命中、缺证据、证据失效、索引失败、变更命中 | GovernanceItem、ReviewTask、AuditEvent | 知识/Evidence 更新、重新索引 |

## 核心用户路径

### 路径一：研发任务上下文包

```text
研发成员选择项目
  -> 输入任务描述
  -> 系统基于 Qdrant 召回代码、知识、接口和规则
  -> 系统基于 Neo4j 补充直接关系和证据链
  -> 用户勾选、排除、排序上下文
  -> 生成上下文包
  -> 复制或导出
  -> 写入审计
```

该路径是首轮 MVP 的主价值路径。

### 路径二：知识沉淀和可信检索

```text
用户从代码事实、检索结果或人工输入创建知识资产
  -> 绑定 Evidence
  -> 项目负责人确认
  -> 更新 Neo4j 知识和证据关系
  -> 标记并重建 Qdrant 索引
  -> 后续检索优先展示已确认知识
  -> 上下文包展示来源、状态和风险
```

草稿知识默认不进入团队级上下文包。待复审知识可以进入检索和上下文包，但必须显示风险提示。

### 路径三：变更影响和治理回流

```text
用户输入文件列表或 diff
  -> 系统命中直接 CodeFact
  -> Neo4j 展开有限深度影响路径
  -> Qdrant 补充语义候选
  -> 输出影响接口、影响知识、风险和测试建议
  -> 命中已确认知识时生成待复审候选或 ReviewTask
  -> 项目负责人处理 ReviewTask
  -> 回写 KnowledgeAsset 或 Evidence 状态
  -> 更新索引和图关系
```

影响分析只能输出候选结论，不能直接把知识改为待复审或失效。最终状态变化必须由项目负责人确认。

### 路径四：低命中治理

```text
用户检索低命中或无结果
  -> 系统脱敏记录 SearchLog
  -> 治理看板展示低命中查询
  -> 项目负责人判断是否补充知识
  -> 创建或更新 KnowledgeAsset
  -> 绑定 Evidence
  -> 重建索引
  -> 后续同类问题命中率提升
```

SearchLog 不应保存密码、token、私钥或完整敏感请求体。

## 闭环依赖规则

- 所有对象必须带 `projectId`，跨项目复用默认禁止。
- Qdrant 检索必须带项目过滤条件。
- Neo4j 图查询必须带项目过滤条件。
- 检索结果必须能追溯到源对象，不能只保留文本片段。
- 上下文包必须保存引用清单，导出必须写审计。
- 知识正文、状态、标签或 Evidence 变化后，必须标记相关索引 stale，并同步更新图关系。
- 任务失败不能清空上一份可用数据。
- 需要人工判断的系统结论必须进入 ReviewTask 或治理项。

## 易断点和兜底要求

| 易断点 | 风险 | 兜底要求 |
|---|---|---|
| 项目同步失败 | 后续分析无输入 | 保留失败原因，允许重试，不创建错误快照 |
| 代码分析部分失败 | 代码事实不完整 | 成功部分可用，失败文件可查看，影响分析展示不确定性 |
| Neo4j 写入失败 | 路径分析不可用 | MySQL 事实仍可用，图写入任务可重试 |
| Qdrant 构建失败 | 检索和上下文召回不可用 | 最近成功索引继续可用，展示新鲜度风险 |
| 检索结果源对象失效 | 上下文或证据引用错误 | 复用前重新校验源对象和权限 |
| Evidence 失效 | 知识可信度下降 | 生成治理项或 ReviewTask，不自动改变知识状态 |
| ReviewTask 被忽略 | 风险被跳过 | 必须记录忽略原因和审计 |

## MVP 闭环边界

首轮 MVP 只要求跑通：

```text
项目接入
  -> 代码事实
  -> Neo4j 基础关系
  -> 知识资产和 Evidence
  -> Qdrant 统一索引
  -> 可信检索
  -> AI 上下文包
  -> 导出审计
```

MVP 不强依赖完整治理看板，但必须预留以下数据：

- SearchLog
- IndexRecord 状态
- AuditEvent
- KnowledgeAsset 状态
- Evidence 有效性
- ContextPackage 引用清单

变更影响、ReviewTask 和治理看板作为第二个核心切片接入，不能反向阻塞 MVP。

## 验收标准

- 用户从一个已授权项目出发，可以完成项目同步、代码分析、知识绑定、索引构建、可信检索和上下文包导出。
- 上下文包中的每条引用都能追溯到项目内源对象。
- 知识或 Evidence 变化后，系统能标记索引 stale，并在更新后影响后续检索结果。
- 变更影响分析命中知识时，只生成候选 ReviewTask，不自动修改知识状态。
- ReviewTask 处理后能回写 KnowledgeAsset 或 Evidence 状态，并触发索引和图关系更新。
- 任务失败时，页面能展示失败原因，并保留上一份可用数据或明确空状态。
- 审计能查询项目创建、成员授权、知识确认、索引构建、上下文导出、影响分析和 ReviewTask 处理记录。
