# 综合研发智能平台功能需求与功能设计

日期：2026-06-17

## 1. 背景与目标

当前系统已经具备 Java 代码分析、知识文档维护、知识资产管理、向量索引、语义检索、API 映射和 Web 控制台等基础能力，但产品结构更像多个工具能力的集合。新规划不以现有菜单为边界，而是将系统重新定义为面向团队协作的综合研发智能平台。

平台以 Web 控制台为主要交付形态，覆盖研发协作、架构治理和 AI 辅助开发。现有代码作为可复用底座，需求和设计按新的产品闭环重新组织。

平台核心目标：

- 帮助团队理解项目代码、接口、模块和调用关系。
- 沉淀可确认、可复审、可追溯的研发知识资产。
- 统一检索代码、文档、知识资产和接口证据。
- 为 Codex、Cursor、Claude Code 等 AI 编程助手生成可信上下文包。
- 基于代码变更初步判断影响范围、风险点和测试建议。

## 2. 产品定位

本系统定位为团队级综合研发智能平台，不是单纯的代码图谱工具，也不是单纯的知识库工具。

系统要回答五类问题：

1. 项目结构、接口、调用链、数据访问事实是什么。
2. 团队确认过哪些业务规则、架构决策、故障经验和开发规范。
3. 当前任务需要哪些可信知识、代码证据和接口上下文。
4. 一次代码变更可能影响哪些接口、模块、知识资产和测试范围。
5. 团队知识是否过期、缺失、低质量或需要复审。

## 3. 设计原则

- 以项目空间为基础隔离数据、权限、索引、任务和知识资产。
- 以研发任务为高频入口，串联检索、上下文包和影响分析。
- 代码分析只提供事实和候选证据，关键知识仍需要人工确认。
- 所有智能推荐必须保留来源、证据和解释，避免黑盒结论。
- 产品应先跑通全场景最小闭环，再持续增强每个能力域的深度。
- 当前实现可复用，但新功能边界不受当前菜单结构限制。

## 4. 目标用户与角色

### 4.1 系统管理员

负责平台初始化、用户管理、项目接入、权限分配、系统配置、任务失败排查和审计查看。

### 4.2 项目负责人

负责项目空间内的成员授权、知识资产确认、复审安排、重要接口说明、业务规则和 ADR 治理。

### 4.3 研发成员

负责日常需求开发、缺陷修复、代码理解、知识补充、可信检索、AI 上下文包生成和变更影响查看。

### 4.4 测试/运维成员

关注接口、业务规则、故障案例、变更影响、测试建议、风险提示和排障知识。

### 4.5 架构治理角色

负责 ADR、开发规范、模块边界、跨项目经验、技术债、知识质量和复审机制。基础实现可以作为项目角色或权限标签存在，不要求复杂审批流。

### 4.6 核心可用版角色收敛

核心可用版采用项目级角色模型：系统管理员、项目负责人、项目成员、只读成员。研发成员默认映射为项目成员；测试/运维成员可按项目需要映射为项目成员或只读成员；架构治理角色在核心可用版中由项目负责人承担，增强版再扩展为独立权限标签或细粒度角色。

## 5. 一级功能域

系统划分为六个一级功能域。

### 5.1 平台底座

承载账号、角色、权限、团队、项目空间、数据源接入、后台任务、通知、审计和系统配置。

### 5.2 代码智能底座

承载代码仓库接入、代码解析、框架识别、结构提取、接口识别、调用关系、数据访问识别、代码切块、分析报告和图谱。

### 5.3 知识资产中心

承载知识文档、结构化知识资产、模板、标签、负责人、状态流转、确认、复审、证据绑定和版本历史。

### 5.4 统一检索与证据中心

承载代码、文档、知识资产、接口、调用链和故障案例的统一索引、检索、重排、证据链展示和可信度标识。

### 5.5 AI 研发助手工作台

承载研发任务输入、上下文包生成、实现建议、风险提示、测试建议、代码评审辅助、导出格式和来源追溯。

### 5.6 变更影响与治理运营

承载 Git 变更检测、影响范围分析、知识过期检测、复审提醒、质量评分、指标看板、规范检查和风险提示。

## 7. 核心业务闭环

平台的业务闭环按研发工作流组织。每个闭环都应独立可运行，并能通过项目空间、代码事实、知识资产和证据关系串联。

### 7.1 项目接入闭环

系统管理员创建项目，接入 Git 仓库，配置项目负责人、成员和默认路径。项目负责人负责项目内成员、路径、知识和治理配置。系统拉取代码后，项目内的代码事实、知识资产、索引、分析结果和任务记录都按项目隔离。

### 7.2 代码事实生成闭环

用户触发代码分析。系统解析项目代码，提取结构、组件、接口、调用关系和 SQL 线索，生成报告、图谱和结构化代码事实。代码事实可以作为检索对象和知识证据。

### 7.3 知识沉淀闭环

研发成员或负责人创建知识文档和结构化知识资产，绑定代码、接口、SQL、调用链或检索结果作为证据。项目负责人或架构治理角色确认知识，确认后的知识成为高可信资产。

### 7.4 可信检索闭环

用户输入自然语言问题。系统在统一索引中检索代码、知识文档、知识资产和接口证据，区分直接命中与关联证据，展示来源、状态、分数、更新时间和证据链。

### 7.5 AI 上下文闭环

用户输入开发任务。系统召回相关知识、代码、接口、调用链、故障案例和规范，按可信度和任务相关性组织成上下文包。用户可以人工勾选、排除和复制给 AI 编程工具。

### 7.6 变更影响闭环

核心可用版中，用户输入文件列表或 diff。系统基于代码事实、接口映射、调用链和知识证据关系，给出影响接口、影响知识、风险点和测试建议。命中的知识资产可进入待复审流程。commit、分支对比和 PR/MR 范围属于增强版输入。

### 7.7 治理运营闭环

系统持续统计知识质量、复审状态、接口覆盖、索引状态和任务执行情况。项目负责人和治理角色可以根据看板发现缺失知识、过期知识、接口风险和低命中查询，并回流到知识补充、复审和项目配置。

## 8. 核心对象

### 8.1 Project

项目空间。包含项目名称、Git 地址、分支、仓库路径、默认代码路径、知识库路径、索引配置、成员和项目状态。

### 8.2 RepositorySnapshot

仓库快照。记录项目某次分析所依据的分支、commit、拉取时间、文件范围和分析基线。

### 8.3 CodeFact

代码事实。包括类、方法、字段、组件、接口、SQL、调用关系、模块依赖等结构化信息。

### 8.4 KnowledgeAsset

结构化知识资产。包括业务规则、ADR、故障案例、接口说明、开发规范、领域术语和模块说明。

### 8.5 Evidence

证据。可以指向文件、类、方法、接口、SQL、调用链节点、索引记录、知识文档片段或外部文档。

### 8.6 IndexRecord

索引记录。表示代码、知识文档、知识资产、接口或调用链的检索单元，包含文本、向量和 metadata。

### 8.7 SearchResult

检索结果。包含命中对象、分数、来源、状态、证据数量、关联结果和解释信息。

### 8.8 SearchLog

检索日志。记录用户查询、命中数量、最高分、过滤条件、是否生成上下文、是否被标记为低命中，以及脱敏后的查询摘要。

### 8.9 ContextPackage

AI 上下文包。包含任务描述、被选中的知识、代码、接口、风险点、测试建议、导出格式和来源追溯。

### 8.10 ChangeAnalysisTask

变更影响分析任务。包含变更输入、影响代码、影响接口、影响知识、测试建议、风险提示和置信度。

### 8.11 ReviewTask

知识复审任务。由用户手动创建，或由代码证据变化、知识过期、质量检查触发。

### 8.12 AuditEvent

审计事件。记录登录、权限变更、项目操作、知识确认、任务执行、上下文导出、敏感操作和系统配置变更。

## 9. 可信度与证据规则

- 已确认知识优先展示为可信知识。
- 待复审知识可以参与检索和上下文包，但必须明确标识风险。
- 草稿知识默认不进入团队级上下文包，可由作者本人显式选择。
- 没有证据的知识可以保存为草稿，但不建议直接确认。
- 代码证据失效或变更后，关联知识应被标记为待复审候选。
- 知识资产正文、状态或证据变化后，必须标记相关 Qdrant 索引和 Neo4j 图关系为待更新，更新成功后再解除新鲜度风险。
- 检索结果复用不能依赖临时前端状态，必须使用可追溯的 `sourceType`、`sourceId`、`chunkId` 或后端保存的检索结果快照。
- ReviewTask 被处理后必须回写 KnowledgeAsset 或 Evidence 的状态；无法回写时必须保留处理说明和审计记录。
- 低命中查询必须形成脱敏 SearchLog，治理看板只能展示用户有权限项目内的汇总和明细。
- 检索、接口映射、影响分析和上下文推荐必须解释推荐原因。
- 用户可以手动修正映射、绑定证据、排除上下文、确认或驳回影响建议。

## 10. AI 上下文包边界

AI 上下文包用于组织可信上下文，不负责自动写代码。

上下文包应包含：

- 任务描述和用户约束。
- 相关已确认知识资产。
- 相关待复审知识及风险标识。
- 相关代码片段、接口、调用链和 SQL 线索。
- 相关开发规范、故障案例和测试建议。
- 来源追溯和生成时间。

基础能力必须支持 Markdown 或纯文本复制格式。增强能力可扩展为 Codex、Cursor、Claude Code 专用格式、API 或 MCP。

## 11. 变更影响分析边界

变更影响分析定位为影响初判，不承诺完整静态分析准确性。

基础能力应支持：

- 核心可用版基于文件列表或 diff 输入；commit 输入属于增强版。
- 识别变更文件对应的类、方法、接口和模块。
- 查找引用这些代码证据的知识资产。
- 列出可能受影响接口和测试建议。
- 标注置信度和不确定性。

当前产品边界不要求：

- 完整跨语言调用图。
- 复杂运行时依赖推断。
- 自动生成最终测试计划。
- 自动判断业务规则冲突。

## 12. 交付范围与优先级

本系统不按阶段拆分需求，而是按“核心闭环必备能力、增强能力、当前产品边界”组织交付范围。核心闭环必备能力用于保证平台可以端到端运行；增强能力用于提升协作治理、工程集成和企业化深度；当前产品边界用于防止需求无限扩张。

### 12.1 核心闭环必备能力

核心闭环必备能力需要覆盖：

- 多用户登录、角色权限、项目授权。
- Git 项目接入、更新、项目空间隔离。
- Java/Spring/MyBatis/前端 API 的基础代码事实提取。
- 知识文档和结构化知识资产管理。
- 知识确认、待复审、归档等基础状态流转。
- 代码、知识文档、知识资产的统一索引。
- 可信检索，结果区分代码、文档、知识资产，并展示状态和证据。
- AI 上下文包，用户输入任务后生成可复制上下文。
- 变更影响初判，基于文件 diff 或指定文件列出影响接口、知识资产和测试建议。
- 基础任务状态，展示分析、索引、上下文生成和影响分析的执行结果与失败原因。

### 12.2 增强能力

增强能力在核心闭环稳定后持续扩展，包括：

- 知识评审流、复审提醒和知识版本历史。
- 变更自动标记知识待复审。
- 知识质量评分、规范模板和标签治理。
- 项目、团队、知识质量和使用情况看板。
- GitHub/GitLab/Gitea PR/MR 集成。
- PR 影响分析、Hook 或 CI 自动索引、代码评审辅助。
- 测试范围推荐和变更风险报告。
- 多项目横向知识检索。
- 统一向量库、重排服务和企业级模型配置。
- 权限审计、敏感信息治理、MCP/API 集成。
- 多语言、多框架扩展。
- 跨项目经验复用和组织级知识地图。

### 12.3 当前产品边界

当前产品边界不要求：

- 完整图数据库平台或组织级知识图谱治理系统。
- 全语言、全框架深度静态分析。
- 自动修改代码或自动提交代码。
- 复杂多级审批流。
- 企业多租户计费。
- 完全自动的知识真假判断。
- 将影响分析结果包装成绝对确定结论。

### 12.4 核心可用版与增强版功能口径

第 6 章是全量功能地图，不代表所有能力都进入核心可用版。核心可用版只实现能支撑端到端闭环的项目内能力；增强版在核心闭环稳定后扩展。

| 功能组 | 核心可用版 | 增强版 |
|---|---|---|
| 项目与组织 | 多项目列表、项目创建、项目配置、项目成员授权、项目默认隔离 | 团队/组织管理、跨项目访问策略、项目模板、组织级知识地图 |
| 权限 | 系统管理员、项目负责人、项目成员、只读成员 | 细粒度模块权限、操作权限矩阵、研发/测试/运维/架构治理独立权限标签 |
| 数据源 | Git 仓库、默认代码路径、知识文档目录 | 数据库元信息、接口文档源、外部文档源、定时同步、增量同步 |
| 代码事实 | Java/Spring/MyBatis/前端 API 基础事实、基础调用关系、Neo4j 项目图写入、基础邻居/路径查询、分析报告和图数据导出 | 深度跨语言调用图、模块规则配置、复杂路径查询、交互式图谱页面 |
| 接口资产 | 通过 `api_doc` 知识资产承载接口说明，API 映射结果可作为证据 | 独立 InterfaceAsset、参数级映射、错误码治理、接口负责人流转 |
| 知识资产 | 草稿、已确认、待复审、已归档、证据绑定、基础模板 | 版本历史、审批流、质量评分、重复和冲突检测、标签治理 |
| 索引检索 | 全量索引、Qdrant 默认向量库、来源过滤、可信状态、证据链、生成知识资产草稿 | 增量索引、失效删除、重排模型、收藏、跨项目检索、组合查询增强 |
| AI 研发助手 | 生成 Markdown/纯文本上下文包、风险和测试建议、引用追溯、导出审计 | 内置 AI 问答、实现建议自动生成、多轮会话、MCP/API 对接 |
| 变更影响 | 文件列表和 diff 输入、候选影响、ReviewTask、测试建议 | commit/分支对比、PR/MR 集成、CI 自动触发、风险报告 |
| 治理运营 | 知识状态、缺证据、待复审、ReviewTask、低命中、索引/分析失败、审计 | 低质量评分、重复/冲突知识、接口覆盖率、规范覆盖率、趋势报表 |
| 安全与导出 | 权限校验、路径隔离、上下文导出审计、基础敏感信息过滤 | 企业级敏感信息治理、导出策略审批、外部 API/MCP 安全策略 |

## 13. 验收标准

### 13.1 项目接入

- 系统管理员可以创建项目并拉取 Git 仓库，项目负责人可以管理本项目配置和成员。
- 项目数据、知识资产、索引和分析结果按项目隔离。
- 普通用户只能访问被授权项目。

### 13.2 代码分析

- 用户可以对项目执行代码分析。
- 系统可以产出 Java 结构、Spring endpoint、组件、SQL 线索、前端 API 调用和分析报告。
- 分析结果可以被检索和绑定为知识证据。

### 13.3 知识资产

- 用户可以创建、编辑、删除知识文档和结构化知识资产。
- 知识资产支持类型、标签、负责人、状态、复审日期和证据。
- 项目负责人或治理角色可以确认知识、标记待复审或归档。

### 13.4 统一检索

- 用户可以按自然语言检索代码、知识文档和知识资产。
- 结果展示来源、类型、可信状态、分数和关联证据。
- 已确认知识在结果中优先突出，待复审知识明确提示风险。

### 13.5 AI 上下文包

- 用户可以输入研发任务并生成上下文包。
- 上下文包包含相关知识、代码证据、接口、风险点和测试建议。
- 上下文包可以复制，并能追溯引用来源。

### 13.6 变更影响初判

- 用户可以在核心可用版输入变更文件或 diff；commit 输入属于增强版。
- 系统列出可能影响的代码、接口、知识资产和测试范围。
- 系统标注不确定性，不把候选影响伪装成确定结论。

### 13.7 治理运营

- 系统可以展示知识数量、状态分布、复审风险、索引状态和最近任务。
- 系统可以定位缺少证据、待复审、过期或低质量的知识资产。
- 关键操作可以追踪操作人、时间、对象和结果。

### 13.8 集成扩展

- 系统可以在 MySQL、向量数据库和对象存储基础上，为重排服务、PR/MR 集成和 MCP/API 提供扩展边界。
- 增强能力上线时不得破坏项目隔离、权限校验和来源追溯。

## 14. 非功能需求

### 14.1 安全

- 所有项目路径、知识库路径和文件操作必须限制在配置的工作区内。
- 所有业务 API 必须校验登录状态和项目权限。
- 管理员能力必须做后端校验。
- AI 上下文包导出前应执行权限校验和敏感信息过滤。
- 关键操作需要记录审计日志。

### 14.2 数据隔离

- 项目代码、知识资产、索引、任务、分析结果和上下文包必须按项目隔离。
- 跨项目检索只在明确授权后开放。

### 14.3 任务可靠性

- 长耗时操作应进入后台任务或具备明确 loading 状态。
- 任务失败应保存错误信息，允许用户重试。
- Git、分析、索引、模型调用失败时，系统应允许用户继续使用已有数据，并提示数据新鲜度风险。

### 14.4 性能

- 平台应支持中小型项目验证，并为更大项目保留扩展空间。
- 索引和检索需要支持分页、来源过滤和类型过滤。
- 企业化部署可以迁移到统一向量库和重排服务。

### 14.5 可观测性

- 系统应展示任务状态、索引数量、失败原因、耗时、存储状态和模型配置状态。

### 14.6 兼容与迁移

- 现有 JSON 状态文件和 JSONL 向量索引只作为迁移来源或开发调试格式，不作为目标产品存储后端。
- `.analzer_projects` 保留为项目工作区，用于项目仓库、分析工作目录和可再生成产物。
- 迁移到 MySQL、向量数据库和对象存储时，应提供迁移脚本和失败回滚策略。

## 15. 技术选型与架构设计

### 15.1 总体架构

系统采用 Web 控制台、Flask API、分析引擎、状态存储、向量数据库、图数据库、对象存储和项目工作区分层架构。

```text
Vue Web Console
  -> Flask API
    -> Analyzer Service
      -> Java Analyzer / API Mapper / Call Graph / KB Loader
      -> State Store: MySQL
      -> Vector Store: Qdrant
      -> Graph Store: Neo4j
      -> Artifact Store: object storage or file service
      -> Project Workspace: .analzer_projects repositories and analysis workspace
```

各层职责如下：

- 前端负责项目操作、知识维护、分析触发、检索交互、上下文包编辑和影响分析展示。
- 后端 API 负责权限校验、参数校验、路径隔离、任务编排、错误转换和响应格式统一。
- 分析引擎负责从代码、文档和知识资产中生成结构化事实、证据和可检索 chunk。
- 状态存储负责用户、项目、授权、知识资产、模板、任务、审计和上下文包记录的持久化。
- 向量数据库负责语义检索、来源过滤、metadata 查询和检索结果排序。
- 对象存储或文件服务负责保存分析报告、导出文件、上下文包快照和大体积分析产物。
- 项目工作区负责保存每个项目的仓库代码、分析工作目录和可再生成临时产物。

### 15.2 前端技术选型

| 方向 | 选型 | 用途 |
|---|---|---|
| UI 框架 | Vue 3 | 构建单页 Web 控制台 |
| 构建工具 | Vite | 本地开发、构建和代理 API |
| 组件库 | Element Plus | 表单、表格、对话框、菜单、通知和状态组件 |
| 图标 | @element-plus/icons-vue | 菜单、按钮和状态图标 |
| 状态组织 | Composition API、`useAnalyzerConsole.ts` | 组织项目、知识、检索、分析和账号状态 |
| Markdown | markdown-it | 知识文档预览、上下文包预览和报告展示 |
| 基础关系展示 | JSON 节点边结构 | 核心可用版用于展示 Neo4j 查询返回的裁剪后关系数据 |
| 图谱渲染 | Cytoscape.js | 增强版用于调用关系、接口关系、模块依赖和知识关系的交互式展示 |
| 图数据交换 | JSON 节点边结构 | 在后端 Neo4j、基础关系展示和增强版图谱组件之间传递图数据 |

前端实现原则：

- 页面共享状态集中在 `web/frontend/src/composables/useAnalyzerConsole.ts`。
- 可复用分析器 UI 放在 `web/frontend/src/components/analyzer`。
- 共享类型放在 `web/frontend/src/types.ts`。
- 新增页面优先复用 Element Plus 组件和现有布局风格。
- 代码分析、向量索引、语义检索、上下文包和影响分析的结果状态应保持解耦，避免一个工作台操作意外覆盖另一个工作台结果。

### 15.3 后端技术选型

| 方向 | 选型 | 用途 |
|---|---|---|
| 运行时 | Python 3.10+ | 分析器、API 服务和命令行工具统一语言 |
| Web 框架 | Flask 3 | 提供 Web API 和会话能力 |
| 路由组织 | Flask Blueprint | 将 API 路由集中在 `web/backend/routes.py` |
| 业务层 | `AnalyzerService` | 统一承载项目、分析、知识、索引和检索业务逻辑 |
| 错误模型 | `APIError` | 面向用户返回稳定错误信息和状态码 |
| 参数校验 | `web/backend/validation.py` | 统一解析 JSON 请求和基础校验 |
| 路径安全 | `web/backend/paths.py` | 限制项目路径、知识库路径和文件操作范围 |
| MySQL 访问 | PyMySQL | 主要状态存储后端 |

后端实现原则：

- API 路由只负责请求解析、登录态获取和调用服务层。
- 业务逻辑放在 `web/backend/services.py`，避免路由层直接操作文件、索引或数据库。
- 所有项目级 API 必须通过项目授权校验。
- 所有文件路径必须通过 workspace/project root 校验，禁止直接信任前端路径。
- 目标产品架构以 MySQL 作为系统状态存储；现有 JSON 状态文件仅作为迁移来源或开发调试格式。

### 15.4 分析引擎技术选型

| 方向 | 选型 | 用途 |
|---|---|---|
| Java 解析 | tree-sitter、tree-sitter-java | 提取 Java 语法结构和源码位置 |
| 数据模型 | Python dataclass | 表达类型、方法、字段、endpoint、SQL、调用点和 chunk |
| 代码切块 | `chunker.py` | 生成适合向量检索的代码事实 chunk |
| 知识库加载 | `kb_loader.py` | 加载 Markdown、TXT、RST、AsciiDoc 并切块 |
| API 映射 | `api_mapper.py` | 匹配前端调用和后端 Spring endpoint |
| 调用关系 | `call_graph.py` | 生成类依赖、方法调用和组件关系 |
| SQL 线索 | `sql_flow.py` | 提取 MyBatis SQL 和数据访问线索 |
| 命令行 | `cli.py` | 支持分析、报告、索引、查询和导出 |

分析引擎实现原则：

- 代码分析输出应保持结构化，避免只生成不可复用的文本报告。
- 所有可引用对象应尽量包含文件路径、符号名称、源码位置和 metadata。
- 代码事实、知识文档和知识资产 chunk 应使用兼容 metadata，便于统一检索和上下文包生成。
- 核心可用版图谱能力必须把项目内 CodeFact、接口、SQL、调用关系、Evidence 和 KnowledgeAsset 的基础关系写入 Neo4j。
- 核心可用版只要求基于 Neo4j 提供项目内基础邻居查询、有限深度路径查询和裁剪后的 JSON 节点边结构导出。
- Cytoscape.js 交互式探索、复杂路径分析、跨项目图谱和图节点直接加入上下文包属于增强版。
- 影响分析只输出候选影响和置信度，不输出绝对结论。
- 多语言扩展应通过新增解析器和 chunk 构建器实现，不应破坏 Java 分析器的稳定接口。

### 15.5 存储技术选型

| 方向 | 选型 | 用途 |
|---|---|---|
| 关系存储 | MySQL | 用户、项目、授权、知识资产、模板、任务、审计、上下文包记录和分析记录 |
| 项目工作区 | `.analzer_projects` | 项目仓库 clone/pull、分析工作目录和可再生成产物 |
| 对象存储或文件服务 | S3 兼容对象存储、NAS 或受控文件服务 | 分析报告、导出文件、上下文包快照和大体积分析产物 |
| 分析结果格式 | JSON、Markdown | 保存分析事实、报告和图谱输出 |
| 向量数据库 | Qdrant | 统一语义索引、metadata 过滤、相似度检索和运维 |
| 图数据 | Neo4j | 核心可用版保存项目内 CodeFact、接口、SQL、Evidence、KnowledgeAsset、ContextPackage 和 ChangeTask 的基础关系 |

存储设计原则：

- 项目代码、知识资产、索引、任务、分析结果和上下文包必须按项目隔离。
- `.analzer_projects` 只承担项目仓库、分析工作目录和可再生成产物，不承担系统业务状态存储。
- MySQL 模式应覆盖核心状态对象，避免依赖本地 JSON 作为事实来源。
- 向量记录必须包含来源类型、对象 ID、项目 ID、文件路径、状态、更新时间和可追溯 metadata。
- 向量数据库是产品检索能力的目标存储；JSONL 仅作为迁移、导入导出或开发调试格式。
- Neo4j 是核心可用版图关系后端，必须按 `projectId` 和 `snapshotId` 隔离图节点和边。
- Neo4j 不替代 MySQL 业务状态存储，也不替代 Qdrant 语义检索；MySQL 保存事实状态，Qdrant 负责语义召回，Neo4j 负责关系追踪和影响路径。

### 15.6 检索与 AI 上下文技术选型

| 方向 | 选型 | 用途 |
|---|---|---|
| 默认 embedding | 统一 embedding 服务或 SentenceTransformers 服务 | 生产语义检索 |
| 开发调试 embedding | HashingEmbedder | 无模型依赖的本地开发和测试 |
| 索引结构 | 向量数据库 collection schema | 向量、文本、来源、状态、项目和证据 metadata |
| 检索策略 | 语义检索、来源过滤、可信状态排序 | 支撑代码、文档、知识资产统一检索 |
| 上下文包格式 | Markdown、纯文本 | 面向 Codex、Cursor、Claude Code 复制使用 |
| 对外集成 | API、MCP 预留 | 支持 IDE、AI 编程助手和自动化工具接入 |

检索与上下文设计原则：

- 检索结果必须展示来源、类型、可信状态、分数和证据关系。
- 已确认知识优先展示，待复审知识可以使用但必须提示风险，草稿知识默认不进入团队上下文包。
- 上下文包应保留引用来源、生成时间、项目、任务描述和用户约束。
- 上下文包生成只组织证据和建议，不自动修改代码。
- 当模型、索引或外部服务不可用时，系统应允许用户继续使用已有数据，并提示数据新鲜度或质量风险。

### 15.7 工程集成技术选型

| 方向 | 选型 | 用途 |
|---|---|---|
| Git 操作 | git 命令或服务封装 | clone、pull、commit 范围、diff 和分支信息 |
| PR/MR 集成 | GitHub、GitLab、Gitea 预留 | 影响分析、评审辅助和质量门禁 |
| CI/CD | Hook、流水线任务预留 | 自动分析、自动索引和变更风险报告 |
| 外部接口 | REST API | 平台能力对外开放 |
| AI 工具协议 | MCP 预留 | 面向 AI 编程助手暴露检索和上下文能力 |

工程集成设计原则：

- Git、PR/MR 和 CI/CD 集成必须复用项目权限和路径隔离规则。
- PR/MR 影响分析应基于 diff、文件列表、代码事实和知识证据关系生成候选影响。
- CI/CD 中的自动结论必须保留置信度，不得把候选影响作为绝对阻断依据。
- 外部 API 和 MCP 能力必须执行登录、授权、审计和敏感信息过滤。

### 15.8 部署架构

基础服务化部署形态：

```text
Browser
  -> Web frontend
  -> Flask API service
  -> MySQL
  -> Qdrant
  -> Object storage or controlled file service
  -> .analzer_projects project workspace
```

扩展部署形态：

```text
Browser
  -> Web frontend
  -> Flask API service
  -> MySQL
  -> Vector backend
  -> Optional graph backend
  -> Git provider
  -> Model or embedding service
  -> Object storage
  -> .analzer_projects project workspace
```

部署原则：

- 本地开发可以在本机启动 Flask、Vue/Vite、MySQL、向量数据库和项目工作区，但不应把本地 JSON/JSONL 作为目标产品存储。
- 向量存储通过后端抽象接口封装，当前实现优先适配 Qdrant，未来可替换为 Milvus 等其他向量数据库。
- 部署形态可以替换向量数据库、对象存储、模型服务和 Git 集成，但不能改变上层 API 语义。
- 后端配置应通过环境变量或配置文件管理，不把本机绝对路径写入业务数据。
- 运行时目录、生成结果、索引和日志不应提交到版本库。

### 15.9 模块边界

### 15.10 核心数据流

项目接入数据流：

```text
创建项目 -> 校验管理员或项目负责人权限 -> 保存 Project 到 MySQL -> 建立项目工作区 -> clone/pull 仓库 -> 授权成员访问
```

统一索引数据流：



```text
输入研发任务 -> 召回相关知识和代码证据 -> 按可信度和任务相关性排序 -> 用户勾选或排除 -> 生成 Markdown/纯文本上下文包 -> 记录来源追溯
```

变更影响分析数据流：

```text
核心可用版输入文件列表或 diff -> 解析变更范围 -> 匹配代码事实、接口和模块 -> 查找绑定知识资产 -> 生成风险点、测试建议和置信度 -> 标记待复审候选；增强版扩展 commit、分支和 PR/MR 输入
```

## 17. 全局功能流图设计

全局功能流图用于描述平台端到端业务链路、代码与知识结合方式、用户任务执行路径、变更影响回溯路径和治理运营闭环。核心可用版要求使用 Neo4j 保存项目内基础图关系，并提供基础邻居查询、有限深度路径查询和可导出的 JSON 节点边结构，用于基础关系展示、证据追溯和影响分析解释。Cytoscape.js 交互式图谱、复杂路径分析和跨项目图谱属于增强版能力，前后端继续以 JSON 节点边结构作为兼容交换格式。

### 17.1 全局主链路

```text
项目接入
  -> 代码同步
  -> 代码分析
  -> 代码事实入库
  -> 知识沉淀
  -> 证据绑定
  -> Qdrant 索引
  -> 可信检索
  -> AI 上下文包
  -> 变更影响分析
  -> 治理复审
  -> 知识更新
  -> 重建索引
```

主链路说明：

- 项目接入负责建立项目空间、成员授权和 `.analzer_projects` 项目工作区。
- 代码同步负责 clone/pull 仓库并形成 RepositorySnapshot。
- 代码分析负责生成 CodeFact、接口、SQL、调用关系和图数据。
- 知识沉淀负责创建 KnowledgeAsset，并通过 Evidence 绑定代码事实。
- Qdrant 索引负责统一索引代码 chunk、知识文档 chunk、知识资产 chunk 和接口 chunk。
- 可信检索负责同时召回代码、知识和证据链。
- AI 上下文包负责把任务、知识、代码证据、风险和测试建议组织成可复制上下文。
- 变更影响分析负责从文件列表或 diff 反向追踪受影响代码、接口和知识；增强版扩展 commit、分支和 PR/MR 输入。
- 治理复审负责把低质量、过期、低命中和受影响知识回流到知识更新。

### 17.2 代码与知识结合流

```text
CodeFact
  -> Evidence
  -> KnowledgeAsset
  -> Neo4j 图关系
  -> IndexRecord
  -> SearchResult
  -> SearchLog
  -> ContextPackage
```

正向绑定流：

```text
代码分析生成 CodeFact
  -> 用户从代码事实、接口、SQL、调用链或检索结果选择证据
  -> 创建 Evidence
  -> Evidence 绑定到 KnowledgeAsset
  -> KnowledgeAsset 经确认后成为可信知识
  -> 写入 Neo4j 图关系
  -> 知识和代码共同进入 Qdrant
```

反向追踪流：

```text
用户命中代码或输入变更
  -> 匹配 CodeFact
  -> 查找 Evidence
  -> 找到关联 KnowledgeAsset
  -> 展示业务规则、接口说明、ADR、故障案例或开发规范
```

结合规则：

- CodeFact 提供事实，KnowledgeAsset 提供语义，Evidence 建立可追溯关系。
- Evidence 可以绑定文件、类、方法、接口、SQL、调用链、索引记录或外部资料。
- 一个 KnowledgeAsset 可以绑定多个 Evidence，一个 CodeFact 可以关联多个 KnowledgeAsset。
- 检索命中知识时必须展示绑定代码证据；检索命中代码时应展示关联知识资产。
- 绑定关系必须按 `projectId` 隔离，禁止跨项目隐式引用。
- Evidence 新增、删除或目标失效时，必须同步更新 Neo4j 中的证据边，并标记受影响知识资产需要重新索引。
- KnowledgeAsset 的正文、状态、标签或证据变化后，必须更新 MySQL 事实记录、Neo4j 节点属性和 Qdrant 索引记录。

### 17.2.1 检索结果复用闭环

```text
用户发起检索
  -> 生成 SearchLog 和 SearchResult 快照
  -> 用户复制、加入上下文包或生成知识资产草稿
  -> 后端使用 sourceType、sourceId、chunkId 或 resultSnapshotId 重新校验项目权限
  -> 创建 ContextPackage 引用或 KnowledgeAsset/Evidence
  -> 写入审计和必要的 Neo4j 关系
```

检索结果复用规则：

- SearchResult 可以是短期展示对象，但复用时必须能回溯到稳定源对象。
- 生成知识资产草稿时，应保存原始命中片段、源对象 ID、chunkId、检索问题和生成时间。
- 复制检索结果只记录脱敏审计，不保存用户复制后的外部用途。
- 低命中、无结果或用户手动反馈“不相关”的查询应进入 SearchLog，作为治理输入。

### 17.3 用户任务流

```text
输入研发任务
  -> 任务类型识别
  -> 提取模块、接口、文件、关键词和约束
  -> 可信检索召回
  -> 记录 SearchLog
  -> 证据链展开
  -> 可信状态排序
  -> 用户勾选、排除和补充
  -> 生成 AI 上下文包
  -> 写入 ContextPackage 与引用关系
  -> 复制或导出
  -> 记录审计
```

任务类型包括需求开发、缺陷修复、重构、接口变更、排障和代码评审。系统应根据任务类型调整召回重点：需求开发优先召回业务规则、接口和模块说明；缺陷修复优先召回故障案例、相关代码和历史知识；接口变更优先召回前后端映射、接口说明和测试建议。

用户任务流输出：

- 任务摘要和用户约束。
- 已确认知识和待复审知识风险提示。
- 相关代码事实、接口、SQL 和调用链。
- 影响范围、风险点和测试建议。
- 引用来源、生成时间和导出审计。

### 17.4 变更影响流

```text
核心可用版输入文件列表或 diff，增强版扩展 commit
  -> 解析变更文件
  -> 解析变更符号
  -> 匹配 CodeFact
  -> 从 Neo4j 展开接口、SQL、调用关系和模块
  -> 从 Qdrant 补充语义相关候选
  -> 查找 Evidence
  -> 找到关联 KnowledgeAsset
  -> 生成影响代码、影响接口、影响知识和测试建议
  -> 写入 ChangeTask 影响关系
  -> 标记待复审候选
```

变更影响流必须区分确定事实和候选推断：

- 变更文件、变更行和匹配到的 CodeFact 属于事实。
- Neo4j 一跳内路径属于高置信候选；有限深度路径、模块级和语义召回结果属于中低置信候选。
- 调用链上游、间接受影响接口、关联知识和测试建议属于候选影响。
- 候选影响必须带置信度、推荐原因和不确定性说明。
- 系统可以生成待复审候选，但不能自动把知识状态改成待复审，除非有权限用户确认。
- 影响分析完成后，ChangeTask 与受影响 CodeFact、Endpoint、KnowledgeAsset 和 ReviewTask 的关系必须写入 Neo4j。

### 17.5 治理闭环流

```text
低质量知识、过期知识、低命中查询、索引失败、变更命中
  -> 生成治理项
  -> 分配负责人或复审人
  -> 复审知识和证据
  -> 更新 KnowledgeAsset 或 Evidence
  -> 更新 Neo4j 图关系
  -> 重建 Qdrant 索引
  -> 影响之后的检索、上下文包和影响分析
```

治理项来源：

- 知识缺少证据、证据失效、复审日期逾期。
- 可信检索低命中或用户频繁无结果查询。
- 变更影响分析命中已确认知识。
- API 映射发现未匹配接口、方法不一致或接口缺说明。
- 分析任务、索引任务或 embedding 任务失败。

治理闭环输出：

- ReviewTask 或治理待办。
- 知识状态变更记录。
- Evidence 更新记录。
- SearchLog 处理记录。
- Neo4j 图关系更新记录。
- 重新索引任务。
- 审计事件。

ReviewTask 处理规则：

- `resolved` 表示负责人已复审并更新或确认相关 KnowledgeAsset/Evidence；如果知识仍然有效，可以从待复审回到已确认。
- `ignored` 表示负责人明确暂不处理候选影响；系统保留原因，不自动改变 KnowledgeAsset 状态。
- ReviewTask 处理后必须写审计，并根据处理结果触发索引和图关系更新。

### 17.6 图节点设计

| 节点类型 | 说明 | 关键属性 |
|---|---|---|
| Project | 项目空间 | projectId、name、branch、status |
| RepositorySnapshot | 仓库快照 | snapshotId、commitHash、branch、syncedAt |
| File | 源码或知识文件 | filePath、sourceType、language、updatedAt |
| Class | Java 类型 | packageName、className、annotations、filePath |
| Method | 方法或构造器 | methodName、signature、startLine、endLine |
| Endpoint | HTTP 接口 | httpMethod、path、controller、handler |
| SQL | SQL 线索 | sqlType、statement、tableNames、mapper |
| FrontendCall | 前端 API 调用 | method、path、filePath、caller |
| KnowledgeAsset | 知识资产 | assetId、type、title、status、tags |
| Evidence | 证据绑定 | evidenceId、evidenceType、confidence |
| IndexRecord | 索引记录 | vectorId、sourceType、sourceId、status |
| ContextPackage | AI 上下文包 | packageId、taskType、createdBy、createdAt |
| ChangeTask | 变更影响任务 | taskId、inputType、status、confidence |
| ReviewTask | 知识复审任务 | reviewId、status、ownerUserId、dueAt |

### 17.7 图边设计

| 边类型 | 起点 -> 终点 | 说明 |
|---|---|---|
| HAS_SNAPSHOT | Project -> RepositorySnapshot | 项目拥有仓库快照 |
| CONTAINS | Project -> File | 项目包含文件 |
| DECLARES | File -> Class/Method/Endpoint/SQL | 文件声明代码事实 |
| HAS_METHOD | Class -> Method | 类型包含方法 |
| EXPOSES | Method -> Endpoint | 方法暴露 HTTP 接口 |
| CALLS | Method -> Method | 方法调用方法 |
| USES_SQL | Method -> SQL | 方法使用 SQL 线索 |
| FRONTEND_CALLS | FrontendCall -> Endpoint | 前端调用后端接口 |
| EVIDENCES | Evidence -> CodeFact | 证据指向代码事实 |
| SUPPORTS | Evidence -> KnowledgeAsset | 证据支撑知识资产 |
| INDEXES | IndexRecord -> CodeFact/KnowledgeAsset/File | 索引记录来源对象 |
| RETRIEVES | SearchResult -> IndexRecord | 检索结果命中索引记录 |
| INCLUDES | ContextPackage -> KnowledgeAsset/CodeFact | 上下文包引用知识或代码 |
| IMPACTS | ChangeTask -> CodeFact/Endpoint/KnowledgeAsset | 变更任务影响对象 |
| REQUIRES_REVIEW | ChangeTask -> ReviewTask | 变更触发复审候选 |

图边规则：

- 所有节点和边必须包含 `projectId`。
- 跨快照关系必须包含 `snapshotId` 或 commit 范围。
- `CALLS`、`IMPACTS`、`REQUIRES_REVIEW` 等推断边必须包含 `confidence` 和 `reason`。
- 核心可用版依赖 Neo4j 保存项目内基础图数据，并用于关系查询、证据追溯和影响路径初判。
- Qdrant 中的向量数据用于语义召回，Neo4j 中的图数据用于关系追踪，两者通过 `sourceId`、`targetId`、`projectId` 和 `snapshotId` 对齐。
- 增强版使用 Cytoscape.js 渲染时，前端只接收后端裁剪后的子图，避免一次性渲染全项目图。

### 17.8 图谱交互设计

本节定义增强版交互式图谱能力。核心可用版需要基于 Neo4j 支持基础邻居查询、有限深度路径查询、裁剪后的 JSON 节点边数据导出，以及在代码事实、接口关系或分析报告中查看基础关系。

增强版图谱页面应支持以下交互：

- 按节点类型过滤：文件、类、方法、接口、SQL、知识、证据、变更任务。
- 按关系类型过滤：调用、接口暴露、SQL 使用、证据绑定、上下文引用、影响关系。
- 从一个节点展开一跳或多跳上下游。
- 点击节点查看详情，并跳转到代码、知识资产、检索结果或影响任务。
- 高亮已确认知识、待复审知识、候选影响和低置信度关系。
- 将图谱中的节点加入上下文包或绑定为知识证据。
- 导出当前子图为 JSON 节点边结构。

图谱接口应支持：

- 按项目和节点 ID 查询邻居。
- 按关系类型查询路径。
- 按任务 ID 查询影响子图。
- 按知识资产 ID 查询证据子图。
- 按 endpoint 查询前后端映射和实现链路。

## 18. 功能详细需求

### 18.1 项目接入

| 项目 | 要求 |
|---|---|
| 入口 | 项目管理页、系统初始化向导、管理员项目配置入口 |
| 前置条件 | 用户已登录；具备系统管理员权限；Git 地址和分支可访问 |
| 主流程 | 创建项目 -> 填写 Git 地址、分支、默认路径 -> 后端校验权限和路径 -> 建立 `.analzer_projects` 项目工作区 -> clone 或 pull 仓库 -> 保存 Project 到 MySQL -> 授权项目成员 |
| 异常流程 | Git 鉴权失败、分支不存在、项目名重复、工作区路径非法、clone 超时、项目成员不存在 |
| 输出结果 | Project 记录、RepositorySnapshot、项目工作区路径、最近同步状态、审计事件 |
| 权限要求 | 创建项目仅系统管理员可操作；项目归档由系统管理员或项目负责人操作；普通成员只能查看被授权项目 |

项目接入必须展示同步状态、最近 commit、当前分支、仓库路径、项目负责人和成员授权情况。Git 同步失败时，系统应保留已有项目数据，并提示失败原因和可重试入口。

### 18.2 代码事实生成

| 项目 | 要求 |
|---|---|
| 入口 | 代码分析页、项目详情页、同步后自动触发入口 |
| 前置条件 | 项目已创建；用户具备项目访问权限；仓库代码存在于项目工作区 |
| 主流程 | 选择分析范围 -> 校验路径 -> 执行 Java/Spring/MyBatis/前端 API 分析 -> 生成 CodeFact、接口、SQL、调用关系和图数据 -> 写入 MySQL 与 Neo4j -> 保存分析记录 -> 更新报告和索引候选状态 |
| 异常流程 | 代码目录不存在、解析失败、文件编码异常、超大文件跳过、部分语言暂不支持 |
| 输出结果 | CodeFact、RepositorySnapshot、分析报告、Neo4j 图节点边、可导出图数据、分析耗时、失败文件列表 |
| 权限要求 | 项目成员可触发分析；管理员和项目负责人可查看失败详情和清理历史结果 |

代码事实必须能被知识证据、可信检索、上下文包和变更影响分析复用。分析结果应区分“已成功解析”“部分失败”“未支持类型”，不能因为部分文件失败而丢弃整个项目结果。

### 18.3 知识沉淀

| 项目 | 要求 |
|---|---|
| 入口 | 知识资产页、知识文档页、检索结果、分析结果、影响分析结果 |
| 前置条件 | 用户具备项目访问权限；知识类型、标题和正文满足基础校验 |
| 主流程 | 创建知识资产 -> 选择类型、标签、负责人和正文 -> 绑定 Evidence -> 保存为草稿 -> 项目负责人确认 -> 进入可信知识集合 |
| 异常流程 | 标题重复、证据不存在、证据跨项目、状态流转非法、确认人权限不足 |
| 输出结果 | KnowledgeAsset、Evidence、状态记录、审计事件、可索引知识 chunk |
| 权限要求 | 成员可创建和编辑本人草稿；负责人和治理角色可确认、标记待复审或归档 |

知识资产必须支持业务规则、ADR、故障案例、接口说明、开发规范、领域术语和模块说明。已确认知识应优先进入可信检索和上下文包；待复审知识可参与召回，但必须展示风险标识。

### 18.4 可信检索

| 项目 | 要求 |
|---|---|
| 入口 | 可信检索页、AI 上下文包工作台、知识资产绑定证据入口 |
| 前置条件 | 项目已构建索引；用户具备项目访问权限；Qdrant collection 可访问 |
| 主流程 | 输入问题 -> 选择检索范围和过滤条件 -> 查询 Qdrant -> 读取 metadata 和关联证据 -> 按相关性、可信状态和来源组织结果 -> 展示解释和追溯 |
| 异常流程 | 索引未构建、Qdrant 不可用、结果为空、metadata 缺失、用户访问未授权来源 |
| 输出结果 | SearchResult 列表、证据链、来源分布、可加入上下文包的候选项 |
| 权限要求 | 用户只能检索被授权项目；跨项目检索必须显式授权 |

检索结果必须包含来源类型、对象标题、分数、可信状态、更新时间、项目 ID、证据数量和推荐原因。检索结果不能只返回文本片段，必须提供可追溯引用。

### 18.5 AI 上下文包

| 项目 | 要求 |
|---|---|
| 入口 | AI 研发助手工作台、可信检索结果、变更影响分析结果 |
| 前置条件 | 用户输入任务目标；项目已存在可用知识、代码事实或索引结果 |
| 主流程 | 输入任务 -> 召回知识和代码证据 -> 标注可信状态和风险 -> 用户勾选、排除、排序 -> 生成 Markdown/纯文本上下文包 -> 记录引用来源和导出审计 |
| 异常流程 | 召回结果为空、内容超过长度限制、引用对象失效、导出权限不足、敏感信息过滤命中 |
| 输出结果 | ContextPackage、上下文文本、引用清单、风险提示、测试建议、导出记录 |
| 权限要求 | 用户只能引用有权限访问的项目资料；上下文包导出必须记录审计 |

上下文包必须包含任务描述、用户约束、已确认知识、待复审知识风险提示、相关代码证据、接口、SQL 线索、风险点、测试建议和生成时间。上下文包不负责自动写代码。

### 18.6 变更影响分析

| 项目 | 要求 |
|---|---|
| 入口 | 变更影响页、项目详情页、未来 PR/MR 集成入口 |
| 前置条件 | 用户具备项目访问权限；核心可用版输入文件列表或 diff；项目存在分析基线 |
| 主流程 | 输入变更范围 -> 解析变更文件和符号 -> 匹配 CodeFact、接口、模块和调用关系 -> 查找绑定知识资产 -> 生成影响代码、影响接口、影响知识、测试建议和置信度 |
| 异常流程 | diff 格式非法、文件列表为空、分析基线缺失、无法定位符号、调用链不完整；commit 不存在属于增强版 commit 分析错误 |
| 输出结果 | ChangeAnalysisTask、影响清单、风险提示、测试建议、待复审候选知识 |
| 权限要求 | 项目成员可执行影响初判；确认待复审状态需要负责人或治理角色权限 |

影响分析必须使用“候选影响”“可能影响”“置信度”等表达，不得把静态分析结果包装成绝对结论。影响命中的知识资产应支持一键标记待复审候选，但最终状态变更需要有权限用户确认。

### 18.7 治理运营

| 项目 | 要求 |
|---|---|
| 入口 | 治理看板、项目看板、知识质量页、审计页 |
| 前置条件 | 用户具备项目访问权限；管理员可查看系统级指标 |
| 主流程 | 汇总知识状态、索引状态、分析任务、上下文导出、低命中查询和复审风险 -> 展示看板 -> 支持跳转到对应资产或任务 |
| 异常流程 | 指标计算失败、数据源不可用、用户无权查看系统级指标 |
| 输出结果 | 项目看板、知识看板、质量看板、使用看板、审计列表 |
| 权限要求 | 普通成员查看项目内指标；管理员查看系统级指标和审计配置 |

治理运营必须帮助用户发现缺证据知识、待复审知识、过期知识、接口缺说明、索引失败、分析失败和低命中查询，并提供回流到知识补充、复审、重建索引或重新分析的入口。

## 19. 核心数据字段

### 19.1 数据字段总览

| 对象 | 关键字段 |
|---|---|
| Project | id、projectKey、name、gitUrl、branch、workspacePath、defaultCodePath、defaultKnowledgePath、ownerUserId、status、createdAt、updatedAt |
| ProjectMember | projectId、userId、role、status、createdAt、updatedAt |
| RepositorySnapshot | id、projectId、branch、commitHash、commitMessage、syncedAt、analyzedAt、sourcePaths、status |
| CodeFact | id、projectId、snapshotId、factType、filePath、packageName、symbolName、signature、startLine、endLine、metadata |
| KnowledgeAsset | id、projectId、type、title、summary、content、status、tags、ownerUserId、reviewerUserId、reviewDueAt、confirmedAt、sourcePath |
| Evidence | id、projectId、assetId、evidenceType、targetId、filePath、symbolName、startLine、endLine、note、confidence |
| IndexRecord | id、projectId、snapshotId、sourceType、sourceId、chunkId、title、text、vectorId、metadata、status、indexedAt |
| SearchResult | id、projectId、sourceType、sourceId、title、snippet、score、status、updatedAt、evidenceCount、explanation |
| SearchLog | id、projectId、queryHash、querySummary、filters、topScore、resultCount、isLowHit、usedForContext、createdBy、createdAt |
| ContextPackage | id、projectId、taskType、taskText、selectedItems、excludedItems、content、format、createdBy、createdAt、exportedAt |
| ChangeAnalysisTask | id、projectId、snapshotId、inputType、inputContent、changedFiles、matchedCodeFacts、impactedEndpoints、impactedKnowledgeAssets、risks、testSuggestions、confidence、status、createdBy、createdAt、baseCommit、headCommit |
| ReviewTask | id、projectId、assetId、sourceTaskId、reason、status、ownerUserId、dueAt、createdAt、updatedAt |
| AuditEvent | id、projectId、actorUserId、action、targetType、targetId、detail、createdAt、ipAddress |

### 19.2 字段规则

- 所有项目级对象必须包含 `projectId`，并参与权限校验和数据隔离。
- 所有可追溯对象必须包含创建时间或更新时间。
- 所有 Evidence 必须能指向代码、接口、SQL、调用链、知识文档片段、检索结果或外部引用之一。
- Qdrant payload 必须包含 `projectId`、`sourceType`、`sourceId`、`chunkId`、`status`、`filePath`、`title` 和 `updatedAt`。
- Neo4j 节点和边必须包含 `projectId`、`snapshotId`、稳定源对象 ID、关系类型、置信度和更新时间。
- SearchLog 必须脱敏保存，不应记录密码、token、私钥或完整敏感请求体。
- ContextPackage 必须保存引用清单，不能只保存最终文本。
- ChangeAnalysisTask 必须保存输入原文、解析结果、影响结果和置信度，便于复查。
- AuditEvent 不应保存敏感明文；必要时保存脱敏摘要或对象 ID。

## 21. 业务规则与验收用例

### 21.1 业务规则

- 普通用户不能查看、检索、导出或分析未授权项目。
- 管理员能力必须由后端校验，前端隐藏按钮不能作为权限控制。
- `.analzer_projects` 只保存项目仓库和可再生成产物，不保存业务状态事实来源。
- 知识资产从草稿进入已确认状态时，必须记录确认人和确认时间。
- 待复审知识进入检索结果或上下文包时必须展示风险提示。
- 草稿知识默认不进入团队级上下文包，作者显式选择除外。
- Qdrant 中的向量记录必须按项目隔离，查询必须带项目过滤条件。
- Neo4j 中的图节点和边必须按项目与快照隔离，图查询必须带项目过滤条件。
- 变更影响分析必须标注置信度和不确定性，不得输出绝对结论。
- 上下文包导出必须记录审计事件，并保留引用来源。
- 关键任务失败不能清空上一份可用结果，应提示数据新鲜度风险。

### 21.2 验收用例

| 编号 | 场景 | 验收条件 |
|---|---|---|
| AC-01 | 未授权项目访问 | 普通用户访问未授权项目 API 时返回 403，前端不展示该项目数据 |
| AC-02 | 项目接入 | 管理员创建项目后，系统在 `.analzer_projects` 建立工作区，并在 MySQL 保存 Project |
| AC-03 | 代码分析部分失败 | 部分文件解析失败时，系统仍保存成功结果，并展示失败文件和原因 |
| AC-04 | 知识确认 | 项目负责人确认知识后，状态变为已确认，并记录确认人和确认时间 |
| AC-05 | 证据绑定 | 用户从检索结果绑定 Evidence 后，知识资产能展示来源、文件和符号信息 |
| AC-06 | Qdrant 索引 | 索引记录写入 Qdrant 时 payload 包含 projectId、sourceType、sourceId、status 和 updatedAt |
| AC-07 | Neo4j 图关系 | 代码分析后，系统把 CodeFact、接口、SQL、调用关系、Evidence 和 KnowledgeAsset 的基础关系写入 Neo4j，并能按 projectId 查询裁剪后的节点边 |
| AC-08 | 可信检索 | 检索结果按来源、分数和可信状态展示，已确认知识优先突出 |
| AC-09 | 待复审提示 | 待复审知识进入上下文包时，正文中出现风险标识和来源追溯 |
| AC-10 | 上下文导出 | 用户复制上下文包后，系统记录导出审计事件和引用清单 |
| AC-11 | diff 影响分析 | 输入 diff 后，系统输出影响代码、接口、知识资产、测试建议和置信度 |
| AC-12 | 影响不确定性 | 影响分析结果使用候选表达，不把结果描述为确定事实 |
| AC-13 | 治理看板 | 看板能展示待复审数量、索引状态、失败任务和低命中查询 |
| AC-14 | 配置失败 | Qdrant、Neo4j 或 embedding 服务不可用时，系统提示失败原因，并保留已有可用数据 |
| AC-15 | 知识变更同步 | 知识正文、状态或证据变化后，系统标记并更新 Qdrant 索引和 Neo4j 关系，检索结果展示最新状态 |
| AC-16 | ReviewTask 处理 | 项目负责人处理 ReviewTask 后，系统回写知识或证据状态，并记录审计 |
| AC-17 | 低命中治理 | 低命中或无结果查询生成脱敏 SearchLog，并能在治理看板中按项目查看 |
| AC-18 | 审计追踪 | 权限变更、知识确认、上下文导出和任务执行都能在审计记录中查询 |
