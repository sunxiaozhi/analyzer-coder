# 综合研发智能平台详细设计索引

本目录是综合研发智能平台详细设计的唯一维护入口，和 `docs/superpowers/requirements/` 下的需求拆分保持一一对应。

详细设计文档回答“怎么落地”：页面结构、核心对象、API、服务边界、状态流转、数据流、错误处理和验收实现口径。

页面原型位于 `../prototypes/`，用于把需求和设计转化为可评审的页面蓝图。

## 核心对象链路

```text
Project
  -> RepositorySnapshot
  -> CodeFact
  -> GraphNode/GraphEdge
  -> Evidence
  -> KnowledgeAsset
  -> IndexRecord
  -> SearchResultSnapshot
  -> ContextPackage
  -> ChangeAnalysisTask
  -> ReviewTask
  -> AuditEvent
```

## 跨存储一致性原则

- MySQL 保存事实状态。
- Qdrant 保存语义向量和可过滤 payload。
- Neo4j 保存项目内基础关系和有限深度路径。
- 三者通过 `projectId`、`snapshotId`、`sourceType`、`sourceId`、`targetType`、`targetId` 和 `chunkId` 对齐。
- 任一跨模块复用都必须传对象 ID 和类型，不能只传展示文本。

## MVP 设计顺序

1. Project、ProjectMember、RepositorySnapshot 和权限校验。
2. CodeFact、基础 Neo4j 节点边和代码事实查询。
3. KnowledgeAsset、Evidence 和知识状态流转。
4. IndexRecord、Qdrant 构建和可信检索。
5. SearchResult 复用到 ContextPackage。
6. ContextPackage 引用清单和导出审计。
7. ChangeAnalysisTask 和 ReviewTask 候选。
8. Governance 看板和 ReviewTask 处理闭环。

前 6 步构成首轮 MVP；第 7、8 步作为第二个核心切片。

## 维护规则

- 设计文档必须与同编号需求文档保持一致。
- 单功能对象、API、页面和服务细节放在 `02` 至 `08`。
- 跨模块对象流转、状态传播和复用契约放在 `10`。
- 架构、服务边界、错误处理、任务状态、非功能和统一验收放在 `09`。
- 修改 Qdrant、Neo4j、MySQL 对齐方式时，必须同步检查 `05`、`09` 和 `10`。
- 修改 ReviewTask、Evidence 或 KnowledgeAsset 状态流转时，必须同步检查 `04`、`07`、`08` 和 `10`。
- 不再新增总详设文档；全局设计内容应归入 `01`、`09` 或 `10`。
- 页面布局、操作入口和跳转关系放到 `../prototypes/`。
