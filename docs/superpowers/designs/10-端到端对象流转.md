# 端到端对象流转详细设计

## 设计目标

本设计定义跨功能域的对象流转、状态传递和数据一致性规则，保证平台不是多个孤立模块，而是一条可以持续回流的项目内研发智能闭环。

设计重点：

- 每个阶段产出的对象必须有明确消费者。
- MySQL、Qdrant 和 Neo4j 通过稳定键对齐。
- 任务失败、索引过期、证据失效和人工复审都有可恢复路径。
- MVP 可以先跑通可信上下文包，后续自然接入变更影响和治理。

## 总体对象链路

```text
Project
  -> RepositorySnapshot
  -> CodeFact
  -> GraphNode/GraphEdge
  -> Evidence
  -> KnowledgeAsset
  -> IndexRecord
  -> SearchResultSnapshot
  -> ContextPackage
  -> ChangeAnalysisTask
  -> ReviewTask
  -> AuditEvent
```

其中 `AuditEvent` 不是业务主链路的输入，但记录每个关键状态变化和导出行为。

## 跨存储对齐键

| 键 | 使用位置 | 说明 |
|---|---|---|
| `projectId` | MySQL、Qdrant payload、Neo4j 节点边、文件路径校验 | 项目隔离主键，所有项目级对象必填 |
| `snapshotId` | RepositorySnapshot、CodeFact、Evidence、Neo4j 边 | 标识代码分析基线 |
| `sourceType` | IndexRecord、Qdrant payload、SearchResult、Evidence | 标识来源类型，如 `code_fact`、`knowledge_asset`、`knowledge_doc`、`endpoint` |
| `sourceId` | IndexRecord、SearchResult、Evidence、Neo4j 节点 | 稳定回溯到源对象 |
| `chunkId` | IndexRecord、Qdrant payload、SearchResult | 文本切片标识 |
| `targetType` | Evidence、ReviewTask、Neo4j 边 | 标识被关联对象类型 |
| `targetId` | Evidence、ReviewTask、Neo4j 边 | 标识被关联对象 ID |
| `taskId` | 分析、索引、上下文包、影响分析、审计 | 追踪异步任务 |

所有跨模块复用都必须优先使用对象 ID 和类型，不允许只传前端展示文本。

## 对象职责和消费者

| 对象 | 事实来源 | 主要字段 | 消费者 |
|---|---|---|---|
| Project | MySQL | `projectId`、名称、状态、工作区、负责人 | 权限、项目空间、任务、审计 |
| ProjectMember | MySQL | `projectId`、`userId`、角色 | 项目 API 权限校验 |
| RepositorySnapshot | MySQL | `snapshotId`、`projectId`、分支、commit、状态 | 代码分析、Evidence 有效性检查 |
| CodeFact | MySQL | `sourceId`、`projectId`、`snapshotId`、文件、符号、类型、摘要 | Neo4j、Qdrant、Evidence、影响分析 |
| GraphNode/GraphEdge | Neo4j | `projectId`、节点类型、对象 ID、关系类型、置信度 | 图查询、影响路径、上下文解释 |
| KnowledgeAsset | MySQL | `targetId`、状态、类型、标签、正文、复审日期 | Qdrant、Neo4j、上下文包、治理 |
| Evidence | MySQL + Neo4j | `sourceType`、`sourceId`、`targetId`、置信度、有效性 | 知识可信度、上下文引用、治理 |
| IndexRecord | MySQL + Qdrant | `vectorId`、`sourceType`、`sourceId`、`chunkId`、状态 | 检索、上下文召回、索引运维 |
| SearchResultSnapshot | MySQL 或短期缓存 | `resultId`、来源、分数、摘要、过期时间 | 加入上下文包、绑定证据、生成草稿 |
| ContextPackage | MySQL + 对象存储 | 任务描述、引用清单、正文、导出记录 | AI 助手、审计、复盘 |
| ChangeAnalysisTask | MySQL + Neo4j | 输入、解析文件、候选影响、置信度 | ReviewTask、治理、审计 |
| ReviewTask | MySQL + Neo4j | 触发原因、关联对象、状态、处理人 | 知识/Evidence 回写、索引和图更新 |
| AuditEvent | MySQL | 操作人、动作、对象、结果、脱敏摘要 | 审计查询、治理排查 |

## 主链路时序

### 1. 项目接入到代码事实

```text
POST /projects
  -> ProjectService 创建 Project
  -> ProjectMember 绑定负责人
  -> RepositorySnapshot 进入 created
  -> AuditEvent(project_created)

POST /projects/{projectId}/sync
  -> 创建同步任务
  -> Git clone/pull 到项目工作区
  -> RepositorySnapshot 进入 created 或 sync_failed

POST /projects/{projectId}/analysis
  -> CodeAnalysisService 创建分析任务
  -> 生成 CodeFact
  -> RepositorySnapshot 进入 analyzed 或 partial_failed
  -> GraphService 写入 Neo4j 基础节点边
  -> IndexService 标记代码类 IndexRecord stale
```

代码分析完成后，必须触发 Evidence 有效性检查。无法定位的 Evidence 生成治理项或 ReviewTask，不自动改变 KnowledgeAsset 状态。

### 2. 知识资产到可信检索

```text
创建或编辑 KnowledgeAsset
  -> 保存 MySQL
  -> 绑定 Evidence
  -> 写 Neo4j HAS_EVIDENCE / SUPPORTS 关系
  -> 标记相关 IndexRecord stale
  -> AuditEvent(knowledge_changed)

POST /projects/{projectId}/indexes/build
  -> IndexService 读取 CodeFact、知识文档、KnowledgeAsset、Evidence
  -> 分块和 embedding
  -> 写入 Qdrant point，payload 带 projectId/sourceType/sourceId/chunkId/status
  -> 更新 IndexRecord 为 ready
```

索引构建采用新旧切换策略：新索引成功前不覆盖最近成功索引。失败时记录原因，保留上一份可用检索能力。

### 3. 可信检索到上下文包

```text
POST /projects/{projectId}/search
  -> SearchService 校验项目权限
  -> Qdrant 按 projectId 和来源过滤检索
  -> 回读源对象补充状态、证据和解释
  -> 生成 SearchResult
  -> 写 SearchLog
  -> 可选保存 SearchResultSnapshot

POST /projects/{projectId}/context-packages
  -> ContextPackageService 接收任务描述和候选 source refs
  -> 重新校验权限和源对象有效性
  -> 读取引用对象正文和证据链
  -> 生成 ContextPackage
  -> 复制或导出时写 AuditEvent(context_exported)
```

上下文包引用清单必须保存 `sourceType`、`sourceId`、`chunkId`、`status`、`score` 和 `evidenceIds`。导出前必须重新执行权限校验和基础敏感信息过滤。

### 4. 变更影响到治理复审

```text
POST /projects/{projectId}/change-analysis
  -> 解析文件列表或 diff
  -> MySQL 命中直接 CodeFact
  -> Neo4j 展开有限深度关系路径
  -> Qdrant 补充语义候选
  -> 生成 ChangeAnalysisTask
  -> 命中已确认 KnowledgeAsset 时生成 ReviewTask 候选
  -> 写 Neo4j IMPACTS / REQUIRES_REVIEW 关系
  -> 写 AuditEvent(change_analysis_created)

POST /projects/{projectId}/review-tasks/{taskId}/resolve
  -> ReviewTaskService 校验项目负责人权限
  -> 回写 KnowledgeAsset 或 Evidence
  -> 标记 IndexRecord stale
  -> 更新 Neo4j 关系
  -> 写 AuditEvent(review_task_resolved)
```

影响分析只生成候选，不直接改变知识状态。`ignored` 也必须保存原因和审计，不触发知识状态变化。

## 状态传播规则

### RepositorySnapshot

| 状态变化 | 触发动作 |
|---|---|
| `created -> analyzing` | 创建代码分析任务 |
| `analyzing -> analyzed` | 写入 CodeFact、Neo4j 基础关系、标记索引 stale |
| `analyzing -> partial_failed` | 保存成功 CodeFact，记录失败文件，标记风险 |
| `analyzing -> failed` | 保留上一快照可用结果，记录失败原因 |

### KnowledgeAsset

| 状态变化 | 触发动作 |
|---|---|
| `draft -> confirmed` | 记录确认人和确认时间，写审计，允许进入团队级召回 |
| `confirmed -> stale` | 写审计，检索和上下文包展示风险 |
| `stale -> confirmed` | 记录复审人和复审时间，更新索引和图关系 |
| `confirmed/stale -> archived` | 默认从后续召回排除，保留历史引用 |

### IndexRecord

| 状态变化 | 触发动作 |
|---|---|
| `ready -> stale` | 检索结果展示新鲜度风险，允许触发重建 |
| `building -> ready` | 切换到新索引版本 |
| `building -> failed` | 保留最近成功索引，记录失败原因 |

### ReviewTask

| 状态变化 | 触发动作 |
|---|---|
| `open -> resolved` | 回写 KnowledgeAsset/Evidence，标记索引 stale，更新图关系，写审计 |
| `open -> ignored` | 保存忽略原因，写审计，不改变知识状态 |
| `open -> reopened` | 保留历史处理记录，重新进入治理看板 |

## 跨模块接口契约

### 复用检索结果

前端传入：

- `projectId`
- `resultId`，或 `sourceType` + `sourceId` + `chunkId`
- 复用目标：`context_package`、`evidence`、`knowledge_draft`

后端必须：

- 校验用户仍有项目权限。
- 回读源对象确认未归档、未删除、未越权。
- 检查对象状态并在结果中保留风险提示。
- 不能信任前端传回的 `snippet` 作为事实来源。

### 写入 Evidence

Evidence 写入必须同时产生：

- MySQL Evidence 记录。
- Neo4j 证据关系。
- KnowledgeAsset 证据数量或可信提示刷新。
- 相关 IndexRecord stale 标记。
- AuditEvent。

### 上下文包导出

导出必须同时产生：

- ContextPackage 记录。
- 引用清单。
- 可选导出文件。
- AuditEvent。

导出失败不能生成成功审计；如果正文生成成功但导出文件失败，应标记为部分失败并保留失败原因。

## MVP 实现顺序

1. Project、ProjectMember、RepositorySnapshot 和项目权限。
2. CodeFact、基础 Neo4j 节点边和代码事实查询。
3. KnowledgeAsset、Evidence 和知识状态流转。
4. IndexRecord、Qdrant 构建和可信检索。
5. SearchResult 复用到 ContextPackage。
6. ContextPackage 引用清单和导出审计。
7. ChangeAnalysisTask 和 ReviewTask 候选。
8. Governance 看板和 ReviewTask 处理闭环。

前 6 步构成首轮 MVP。第 7、8 步作为第二个核心切片。

## 一致性检查清单

- 任一项目级对象是否都有 `projectId`。
- Qdrant payload 是否包含 `projectId`、`sourceType`、`sourceId`、`chunkId` 和状态。
- Neo4j 节点边是否包含 `projectId`，推断边是否包含 `confidence` 和 `reason`。
- SearchResult 是否能回溯到源对象。
- ContextPackage 是否保存引用清单，而不仅是最终文本。
- KnowledgeAsset 或 Evidence 变化后是否标记索引 stale。
- ReviewTask 处理后是否触发索引和图关系更新。
- 关键操作是否写 AuditEvent。
- 任务失败是否保留上一份可用数据或明确空状态。
